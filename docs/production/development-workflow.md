# 开发流程

## 工作方式

太阳农场采用小步快跑的垂直切片开发方式。每个功能都尽量形成可玩、可测试、可回退的完整闭环，而不是长期停留在半成品系统。

参考项目：https://github.com/donchitos/claude-code-game-studios

## 推荐迭代节奏

1. 定义一条玩家可感知的目标。
2. 写下最小验收清单。
3. 实现玩法逻辑和必要表现。
4. 本地运行并手动游玩。
5. 补充必要测试或调试工具。
6. 更新文档和任务记录。

## 首个里程碑：可玩养鸡切片

验收清单：

- 游戏可以启动到农场地图。
- 玩家可以使用移动端触摸摇杆移动。
- 玩家靠近鸡舍后可以喂鸡。
- 玩家点击下一天后，已喂鸡数量可以转化为鸡蛋。
- 玩家靠近商店后可以买饲料、买鸡、卖鸡蛋和卖鸡。
- 背包数量正确变化。
- 刷新页面后可以读取存档。
- 可以清除存档重新开始。

## 后续里程碑

### M2：经营反馈

- 更完整的商店买卖反馈。
- 金币增长动画。
- 基础音效。
- 鸡动画、鸡舍反馈和产蛋表现。

### M3：农场扩展

- 建造物。
- 更多作物。
- 地图扩展。
- 简单任务。

### M4：生活感

- NPC 和商店营业时间。
- 天气。
- 节日或每日事件。
- 更完整的 UI 和设置页。

## 分支建议

- `main`：稳定可运行版本。
- `feature/*`：单个玩法或系统开发。
- `docs/*`：文档更新。
- `chore/*`：构建、依赖、脚手架等维护工作。

## 提交信息建议

使用简洁动词开头：

- `feat: add chicken farming prototype`
- `feat: add shop transactions`
- `fix: correct egg production calculation`
- `chore: configure vite and phaser`

## 每次功能完成前检查

- 是否能本地启动。
- 是否有明显控制台错误。
- 是否更新相关文档。
- 是否记录新增素材来源。
- 是否避免把玩法数值硬编码在多个地方。

## 风险记录

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| Phaser 4 API 仍在变化 | 升级成本和兼容性风险 | 锁定版本，封装项目内部接口 |
| 素材风格不完全匹配农场主题 | 视觉一致性不足 | 原型期先使用，后续替换或二次绘制 |
| 系统耦合过早变重 | 后续扩展困难 | 保持 Scene、System、Data 分层 |
| 养成数值缺少验证 | 节奏失衡 | 使用配置表并保留调试入口 |
